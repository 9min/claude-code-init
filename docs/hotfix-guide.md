# 핫픽스 대응 가이드

프로덕션 환경에서 긴급 장애가 발생했을 때, 신속하고 체계적으로 대응하기 위한 가이드이다.

---

## 핫픽스 정의 및 사용 기준

### `fix/` vs `hotfix/` 구분

| 구분 | `fix/*` | `hotfix/*` |
|------|---------|------------|
| 대상 | 일반 버그 | 프로덕션 장애·긴급 버그 |
| 기반 브랜치 | `main` 최신 | `main` 최신 |
| 리뷰 프로세스 | 일반 (전체 체크리스트) | 간소화 (핵심 항목만) |
| 배포 우선순위 | 일반 큐 | 즉시 배포 |
| 사후 조치 | 없음 | 포스트모템 필수 (P0/P1) |

### 핫픽스 대상 판단 기준

다음 중 하나 이상에 해당하면 `hotfix/` 브랜치를 사용한다:

- 프로덕션에서 사용자가 **핵심 기능을 사용할 수 없는** 경우
- **데이터 유실** 또는 **보안 취약점**이 발견된 경우
- 결제·인증 등 **비즈니스 크리티컬 기능**이 오작동하는 경우
- 다수 사용자에게 **즉각적인 영향**을 미치는 버그

> 위 기준에 해당하지 않는 일반 버그는 `fix/` 브랜치로 처리한다.

---

## 긴급도 분류

| 레벨 | 이름 | 설명 | 대응 시간 | 예시 |
|------|------|------|-----------|------|
| P0 | Critical | 전체 서비스 중단, 데이터 유실, 보안 침해 | 즉시 (1시간 이내) | 인증 완전 불가, DB 데이터 노출 |
| P1 | Major | 핵심 기능 장애, 다수 사용자 영향 | 4시간 이내 | 결제 실패, 주요 페이지 에러 |
| P2 | Minor | 일부 기능 불편, 소수 사용자 영향 | 24시간 이내 | UI 깨짐, 특정 조건에서만 발생하는 버그 |

> P2는 상황에 따라 일반 `fix/` 브랜치로 처리할 수 있다.

## 핫픽스 워크플로우

### 1. 상황 파악 및 긴급도 판단

- 장애 현상을 확인하고 영향 범위를 파악한다.
- 위 긴급도 분류 기준에 따라 레벨을 결정한다.
- 팀에 장애 상황을 공유한다 (Slack, 이슈 등).

### 2. 브랜치 생성

```bash
git checkout main    # 또는 master (프로젝트의 기본 브랜치에 맞춤)
git pull origin main
git checkout -b hotfix/간결한-설명
# 예: hotfix/auth-token-validation-fix
```

### 3. 최소 범위 수정

- **수정 범위를 최소한으로 유지한다.** 장애 해결에 직접 관련된 변경만 포함한다.
- 리팩토링, 코드 정리 등 장애와 무관한 변경은 포함하지 않는다.
- 근본 원인의 완전한 해결보다 **즉각적인 문제 해결**에 집중한다.

> 근본 원인에 대한 구조적 수정이 필요하면, 핫픽스 배포 후 별도 `fix/` 또는 `feature/` 브랜치에서 진행한다.

### 4. 테스트

- 수정 관련 기존 테스트를 실행한다.
- 수정 사항에 대한 **회귀 테스트를 추가**한다.

```bash
pnpm vitest run --reporter=verbose
```

> P0 긴급 상황에서도 최소한의 테스트는 반드시 실행한다. 테스트 없이 배포하지 않는다.

### 5. PR 생성 및 코드 리뷰

- PR 제목에 `[HOTFIX]` 접두사를 사용한다.
- PR 본문에 다음 정보를 포함한다:
  - 장애 현상
  - 원인 분석
  - 수정 내용
  - 영향 범위
  - 긴급도 레벨

```
🐛 fix: [HOTFIX] 토큰 검증 로직 오류 수정

## 장애 현상
로그인 후 토큰 만료 시 무한 리다이렉트 발생

## 원인
토큰 갱신 로직에서 만료 시간 비교 오류

## 수정 내용
만료 시간 비교를 UTC 기준으로 통일

## 영향 범위
인증이 필요한 모든 페이지

## 긴급도
P0 - 전체 사용자 로그인 불가
```

- 긴급도에 따른 코드 리뷰 프로세스를 적용한다 (아래 섹션 참조).

### 6. 배포

- CI 체크 통과를 확인한다.
- Squash and Merge 후 Vercel 자동 배포를 확인한다.
- 배포 완료 후 모니터링을 시작한다 (아래 섹션 참조).

### 7. 사후 조치

- 배포 후 모니터링 기간 동안 장애 재발 여부를 확인한다.
- P0/P1의 경우 포스트모템을 작성한다 (아래 섹션 참조).
- 근본 원인에 대한 구조적 수정이 필요하면 후속 이슈를 생성한다.

## 코드 리뷰 간소화 기준

### 핫픽스 전용 리뷰 체크리스트

- [ ] 수정이 장애 현상을 해결하는가?
- [ ] 수정 범위가 최소한인가? (관련 없는 변경 포함 여부)
- [ ] 새로운 보안 취약점을 도입하지 않는가?
- [ ] 기존 기능에 사이드 이펙트가 없는가?
- [ ] 수정에 대한 테스트가 포함되었는가?
- [ ] RLS 정책 변경이 있다면 검증되었는가?

> 전체 코드 리뷰 체크리스트는 [코드 리뷰 체크리스트](code-review-checklist.md)를 참조한다.

### 긴급도별 리뷰 프로세스

| 긴급도 | 리뷰어 수 | 리뷰 방식 | 비고 |
|--------|-----------|-----------|------|
| P0 | 1명 (구두 확인 가능) | 즉시 리뷰, 화면 공유 가능 | 배포 후 추가 리뷰 수행 |
| P1 | 1명 | PR 리뷰 (4시간 내 응답) | - |
| P2 | 1명 | 일반 리뷰와 동일 | - |

> **주의:** P0에서 구두 확인으로 머지한 경우, 배포 후 **24시간 이내에 반드시 정식 코드 리뷰를 완료**한다.

## 배포 후 모니터링 및 검증

### 배포 직후 검증 체크리스트

- [ ] 프로덕션 URL 접속 및 정상 동작 확인
- [ ] 수정 대상 기능 직접 테스트
- [ ] 브라우저 콘솔 에러 확인
- [ ] Vercel 배포 로그 정상 확인
- [ ] Supabase 대시보드에서 에러 로그 확인 (DB 관련 수정 시)

### 긴급도별 모니터링 기간

| 긴급도 | 집중 모니터링 | 관찰 기간 |
|--------|-------------|-----------|
| P0 | 배포 후 2시간 | 48시간 |
| P1 | 배포 후 1시간 | 24시간 |
| P2 | 배포 후 30분 | 12시간 |

## 롤백 연계

핫픽스 배포 후 새로운 장애가 발생하면, 핫픽스 재수정보다 **먼저 롤백을 고려**한다.

### 핫픽스 실패 시 대응 절차

1. 핫픽스 배포 후 새 장애 발생을 확인한다.
2. 즉시 Vercel 이전 배포로 롤백한다.
3. 원인을 분석한 후 새 핫픽스 브랜치를 생성한다.
4. 수정 후 재배포한다.

> 롤백 방법은 [CI/CD 가이드의 롤백 전략](cicd-guide.md#롤백-전략)을 참조한다.

## 포스트모템

P0/P1 핫픽스 완료 후 반드시 포스트모템을 작성한다.

### 포스트모템 템플릿

```markdown
## 포스트모템: [장애 제목]

### 요약
- **발생 시각**: YYYY-MM-DD HH:MM
- **감지 경로**: (모니터링/사용자 제보/내부 발견)
- **긴급도**: P0/P1
- **영향 범위**: (영향받은 사용자 수, 기능)
- **해결 시각**: YYYY-MM-DD HH:MM

### 타임라인
| 시각 | 이벤트 |
|------|--------|
| HH:MM | 장애 발생 |
| HH:MM | 장애 감지 |
| HH:MM | 대응 시작 |
| HH:MM | 핫픽스 배포 |
| HH:MM | 정상 확인 |

### 근본 원인
(장애가 발생한 근본적인 원인을 기술한다)

### 재발 방지 대책
- [ ] (구체적인 재발 방지 조치 항목)

### 관련 링크
- PR: #000
- 이슈: #000
```
